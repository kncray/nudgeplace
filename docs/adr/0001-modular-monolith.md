# ADR 0001: Django 모듈러 모놀리스 채택

- **Status**: Accepted
- **Date**: 2026-07-08
- **근거 원칙**: 헌장 VII(모듈러 모놀리스 우선) · **Spec**: `specs/000-foundation-structure/`

## 맥락

Nudgeplace는 멀티벤더 커머스 플랫폼을 단계적으로 구축하며 학습하는 프로젝트다.
도메인 경계가 아직 코드로 검증되지 않은 시점에 배포 토폴로지부터 쪼개면, 분산
시스템 복잡도(네트워크 경계·부분 실패·분산 트랜잭션·운영 오버헤드)가 도메인
모델링보다 먼저 비용을 청구한다.

## 결정

단일 배포 단위의 **Django 모듈러 모놀리스**로 시작한다.

- 각 비즈니스 도메인은 `apps/<domain>/`의 독립 Django 앱으로 구현한다(원칙 VII).
- 도메인 경계는 관례가 아니라 기계로 강제한다 — [ADR 0002](./0002-boundary-enforcement-tach.md).
- 모놀리스 내부의 도메인 간 서비스 호출은 하나의 DB 트랜잭션을 공유할 수 있다
  (원칙 VIII MAY) — 순수 내부 처리를 최종 일관성으로 구현하도록 강요하지 않는다.

## 검토한 대안

- **처음부터 마이크로서비스**: 경계가 미검증인 상태에서 분산 복잡도를 선지불한다.
  잘 모듈화된 모놀리스가 향후 어떤 분리든 그 전제 조건이다. 기각.
- **비-Django 스택**: 헌장의 기술 제약(Django 모놀리스)과 상충하며, 비-Django
  재작성은 프로토타입/스파이크 한정이다. 기각.

## 트레이드오프

- 단일 배포 단위의 **결합 위험**을 감수한다 — 원칙 VIII의 기계 강제(Tach + CI
  required check)가 이를 상쇄한다.
- 수평 확장 단위가 앱 전체라는 제약을 수용한다 — 도메인별 독립 확장은 서비스
  추출 이후의 문제로 미룬다.

## 진화 경로

모놀리스는 시작 상태이지 상한이 아니다. 경계가 안정적으로 검증되고 **측정된
운영상의 필요**가 문서화되면(Structure Decision + 신규 ADR), 해당 도메인을 별도
Django 서비스로 추출한다(원칙 VII — "필요가 입증되기 전까지 도입 금지").
