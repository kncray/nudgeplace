# ADR 0003: 공유 커널 단방향 의존

- **Status**: Accepted
- **Date**: 2026-07-08
- **근거 원칙**: 헌장 VIII(명확한 도메인 경계) · **Spec**: `specs/000-foundation-structure/` (BR-3)

## 맥락

횡단 primitive(Money·도메인 이벤트 계약·멱등키·감사·행위자 역할 — Spec 001)가
놓일 공유 위치가 필요하다. 공유 위치의 고전적 실패 모드는 커널이 도메인을
역참조하기 시작하면서 "모든 것이 모든 것에 의존하는" 잡동사니(common)로
퇴화하는 것이다.

## 결정

`shared_kernel/` 패키지를 공유 커널의 위치로 확정하고 **단방향 의존**을 기계
강제한다: 도메인(`apps.*`)→커널 의존은 허용, 커널→도메인 역의존은 차단
(`tach.toml`에서 `shared_kernel`의 `depends_on = []`, BR-3).

커널은 이 스펙에서 **빈 위치**로 남는다(public symbol 0, SC-003) — 내용물은
Spec 001이 채운다. *"공유 커널"은 DDD의 bounded-context 간 공유 도메인 모델이
아니라 이 프로젝트 내부의 플랫폼 기반 용어다.*

## 빈 커널에서 계약이 공허(vacuous)하지 않은 이유

역의존 계약은 "커널이 import할 도메인"이 존재해야 시험 가능하다. 이를 위해 경계
적합성 픽스처 앱(`apps.conformance_a`·`conformance_b`)을 **도메인 계층으로
등록**하고(FR-005), 메타 테스트가 "커널→픽스처 import" 위반 샘플을 임시 복제본에
주입해 실제 차단됨을 검증한다(SC-002) — 빈 커널에서도 규칙이 살아 있음을 실증한다.

## 검토한 대안

- **양방향 허용(제약 없음)**: 커널이 도메인 편의 코드를 흡수하며 잡동사니화 —
  경계 붕괴의 전형. 기각.
- **커널 없이 도메인 간 직접 공유**: 공통 primitive가 특정 도메인에 갇히고 그
  도메인이 사실상의 커널이 된다. 기각.
- **명칭 `common`/`platform`**: `common`은 잡동사니화를 부르는 이름, `platform`은
  의미 중복 — `shared_kernel`이 의도(공유 + 단방향 규율)를 가장 분명히 표현. 기각.

## 트레이드오프

- 커널에 두기 애매한 코드(두 도메인만 쓰는 헬퍼 등)의 배치 판단 비용을 수용한다
  — 기본값은 "도메인에 두고 services로 노출", 커널 승격은 셋 이상이 쓸 때 검토.
- 빈 패키지를 먼저 두는 선점 비용은 SC-003(심볼 0 기계 단언)으로 통제한다.

## 진화 경로

Spec 001이 Money·이벤트 계약·멱등·감사·역할 primitive를 커널에 채우고, Spec 002의
walking skeleton이 픽스처 앱을 확장해 커널 primitive **사용**을 관통 증명한다
(픽스처의 모델·primitive **정의** 금지는 유지 — 정의/사용 분리).
