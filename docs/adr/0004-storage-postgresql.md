# ADR 0004: PostgreSQL 저장소 채택

- **Status**: Accepted
- **Date**: 2026-07-10
- **근거 원칙**: 헌장 XIII(멱등성) · XVII(근거 기반 검증) · **Spec**: `specs/001-shared-kernel-primitives/`

## 맥락

Spec 001은 프로젝트 최초의 영속 primitive를 도입한다. 멱등 실행 기록은
`(namespace, scope, key)` 유니크 제약, 행 잠금, 조건부 갱신, 트랜잭션 롤백 의미에
의존한다. 감사 사실도 상태 변경과 같은 트랜잭션에서 기록되어야 한다.

SC-004의 핵심은 "동시 도착에서도 효과는 최대 1회"다. 이 보장은 테스트 데이터베이스의
동시성 의미가 운영 데이터베이스와 달라지면 검증 증거로 쓸 수 없다.

## 결정

Spec 001부터 저장소를 **PostgreSQL 16**으로 확정한다.

- Django 설정은 PostgreSQL 접속 정보를 환경변수로 받는다.
- 로컬 개발은 `docker-compose.yml`의 postgres 서비스로 실행한다.
- CI `quality-gate`는 GitHub Actions postgres 서비스 컨테이너를 사용한다.
- Python 드라이버는 `psycopg[binary] 3.2+`를 사용한다.

## 검토한 대안

- **SQLite 유지 후 배포 시 PostgreSQL 전환**: SQLite의 단일 작성자 잠금은 PostgreSQL의
  유니크 충돌 대기·행 잠금 의미와 다르다. 동시성 테스트의 대표성을 잃고, 나중에
  저장소 전환 비용을 다시 낸다. 기각.
- **MySQL**: 사용 가능한 선택지지만 현재 Django 생태와 PostgreSQL의 트랜잭션·제약
  기능 쪽이 이 스펙의 검증 요구에 더 직접적으로 맞는다. 기각.
- **애플리케이션 레벨 락 또는 메모리 저장소**: 프로세스 재시작·다중 프로세스·운영
  DB 불변식을 다루지 못한다. 기각.

## 트레이드오프

- 로컬 개발에 Docker 의존이 생긴다.
- CI 시간이 소폭 늘어난다.
- 스캐폴딩 단계보다 설정 복잡도가 올라간다.

이 비용은 멱등성·감사 기록의 운영 대표성을 얻기 위한 비용으로 수용한다.

## 진화 경로

운영 배포 시에는 매니지드 PostgreSQL로 그대로 이행한다. 외부 부수 효과와 아웃박스가
도입되는 P6/P12에서도 같은 트랜잭션·제약 의미를 확장해 사용한다.
