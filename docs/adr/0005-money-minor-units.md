# ADR 0005: Money 최소 단위 정수 표현

- **Status**: Accepted
- **Date**: 2026-07-10
- **근거 원칙**: 헌장 V(금액과 재무적 정확성) · **Spec**: `specs/001-shared-kernel-primitives/`

## 맥락

커머스 금액 오류는 조용히 누적되고 정산·환불·감사 단계에서 크게 터진다. 헌장 V는
금액에 부동소수점을 금지하고, 통화·정밀도·반올림 정책을 명시하라고 요구한다.

Spec 001은 모든 미래 도메인이 사용할 Money primitive를 처음 정의한다. 여기서 표현을
잘못 고르면 이후 가격, 할인, 세금, 정산, 환불이 모두 같은 결함을 상속한다.

## 결정

Money 내부 액수는 **통화 최소 단위의 정수(minor units)** 로 표현한다.

- `Money.amount_minor`는 `int`다.
- 통화 정밀도는 `Currency(code, minor_unit_exponent)` 레지스트리가 소유한다.
- 초기 등록 통화는 KRW(지수 0)다.
- Money 생성 경로는 minor-unit `int`를 받고, `float` 직접 입력을 거부한다.
- 정밀도 손실 가능성이 있는 스칼라 곱은 정확 값(**정확 십진 문자열** 또는
  `Fraction`)과 명시적 반올림 정책을 요구한다. `Decimal` 인스턴스는 float
  유래 여부를 객체만으로 판별할 수 없어 **생성·연산 모두 받지 않는다**(같은
  근거의 일관 적용 — 문자열은 내부에서 `Decimal(str)`로 정확 변환).
- 합계 보존 분배는 `Money.allocate()`가 정수 나눗셈과 끝전 귀속 정책으로 처리한다.

## 검토한 대안

- **float**: 헌장 V가 직접 금지한다. 이진 부동소수점 오차와 암묵 반올림은 금액
  primitive의 시작점으로 부적합하다. 기각.
- **Decimal 저장**: 십진 표현력은 좋지만 `Decimal(0.1)`처럼 이미 float 오차를 가진
  값이 들어온 경우 객체만으로 유래를 판별할 수 없다. 스케일 정규화도 별도로
  관리해야 한다. 기각.
- **문자열 금액 저장**: 외부 입력 표현으로는 가능하지만 산술·비교·안분의 내부
  표현으로는 매번 파싱과 정규화가 필요하다. 기각.

## 트레이드오프

- API 사용자는 "원 단위 숫자"가 아니라 "최소 단위 정수"를 의식해야 한다.
- 비정수 배율 계산은 `Decimal`/`Fraction`과 반올림 정책을 명시해야 하므로 호출부가
  장황해진다.

이 마찰은 의도된 것이다. 재무 결과가 달라지는 지점은 흐름별 스펙이 명시적으로
정책을 선택해야 한다.

## 진화 경로

다통화는 통화 레지스트리에 새 통화와 정밀도를 등록하는 방식으로 확장한다. 환율과
자동 환산은 Money primitive 범위가 아니며, 실제 필요가 생기는 스펙에서 별도 도메인
규칙으로 다룬다.
