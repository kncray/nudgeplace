# Nudgeplace 개발 로드맵 (Development Roadmap)

Nudgeplace를 처음부터 프로덕션 수준의 멀티벤더 커머스 플랫폼까지 성장시키기
위한 단계적 개발 계획이다. 이 문서는 [헌장(Constitution)](../.specify/memory/constitution.md)의
원칙을 **의존성 순서대로 학습·구축 가능한 단위로 분해**한 결과다. 각 Phase는
하나의 **마일스톤(milestone)** 이며, 작은 Phase는 스펙(Spec) 1개로, 무거운
Phase는 의존성 순서를 지키는 **여러 개의 스펙**으로 구성된다.

> **위치**: 이 문서는 각 Phase의 `speckit-specify` 진입점이자 Phase 간 의존성
> 참조 문서다. Phase 순서·경계 변경은 이 문서 개정으로만 한다.

---

## 설계 목표 (Goals)

1. **실제 운영 가능한 멀티벤더 커머스 플랫폼**을 완성해 릴리스한다.
2. **학습 순서가 자연스럽도록** 도메인을 단계적으로 확장한다.
3. **후반부 대규모 리팩터링을 방지**하도록 도메인 의존성을 미리 고려한다.

## 로드맵을 관통하는 두 가지 설계 원리

### ① "설계는 먼저, 구축은 나중" (후반 리팩터링 방지의 핵심)

정산·대사·관측성이 요구하는 값과 신호, 그리고 프로모션·벤더 생명주기가
후속 도메인에 미치는 효과는 그것을 *구축*하기 훨씬 전에 *계약(port·상태·이벤트)*
으로 정의되고 *입력값*이 동결·기록되어야 한다. 이를 놓치면 후반에 모든 선행
도메인을 소급 수정해야 한다. 다음 실(thread)들이 여러 Phase를 관통한다.

| 나중에 구축 | 지금부터 심는 계약/입력 (심는 Phase) |
|---|---|
| **Promotion (P8)** | Pricing/Promotion decision **port** + no-op 정책(P5) |
| **Vendor 생명주기 (P11)** | 상태·전이·이벤트 **계약**(P1) → 노출/주문/정산 효과 연결(P2·P5·P10) |
| **정산 (P10)** | 1P 판매자 계정·`vendor_id` 사용 범위 계약(P1), 라인별 위탁/직매입 분류(P5), 프로모션 재원 귀속(P8), 역방향 차감 사실(P9), 과세/면세 구분(P2)·세금 동결(P5) |
| **배송비 안분·재평가 (P9)** | 벤더 소유 배송비 정책·묶음배송 그룹 규칙 정의(P2) → 실시간 미리보기(P4) → 그룹 단위 산정·동결(P5) |
| **대사·관측 운영 (P12)** | 상관관계 ID·조인 키·구조화 신호(P3·P6·P8·P10·P11에서 첫날부터) |

### ② "얇게 관통 → 두껍게 덧입힘"

`P0~P7`로 정방향 거래 척추(카탈로그 → 주문 → 결제 → 이행)를 먼저 완성해
모듈러 모놀리스 경계를 조기 검증하고, `P8~P11`로 Promotion·Reverse·
Settlement·Vendor 생명주기를 그 위에 덧입힌다.

```
P0 커널·경계 → P1 Vendor·Identity(관통 실증) → P2 Catalog → P3 Inventory·예약 → P4 Cart
   → P5 Order(동결) → P6 Payment(ACL) → P7 Fulfillment    [정방향 척추 완성 → ★M1 내부 베타]
   → P8 Promotion → P9 Reverse                            [→ ★M2 제한 파일럿(실거래)]
   → P10 Settlement                                       [재무 도메인 덧입힘]
   → P11 Vendor 생명주기(FSM) → P12 운영 하드닝·Go-live     [완결·운영 진입]
```

### Phase 분할 원칙

- **P0 = 프로젝트 기반 + 아키텍처 검증** (도메인 기능 선점 금지).
- **P1 이후 = 실제 도메인 구현.**
- **Phase = 마일스톤; 내부에 1..N개의 순서있는 스펙**을 둔다. 작은 Phase는
  스펙 1개(P0·P2·P4 등)로, 무거운 Phase(P6·P9·P10·P12)는 여러 스펙으로
  분할하되 **Phase 간 의존성 순서는 유지**한다. 각 Phase의 "구현할 핵심 도메인"에
  스펙 분할 예시를 함께 적는다.
- Inventory(P3)와 Cart(P4)는 병합하지 않는다.
- P0와 P1은 병합하지 않는다.

각 Phase는 **목적 / 구현할 핵심 도메인 / 선행 조건 / 완료 조건 / 다음 Phase
연결 이유**를 명시한다. 괄호 안 로마 숫자는 근거가 되는 헌장 원칙 번호다.
모든 Phase·스펙은 아래 [공통 Definition of Done](#공통-definition-of-done)을
충족해야 한다.

## Phase 0 — Platform Foundation (공유 커널 + 아키텍처 검증)

- **목적**: 해당 도메인이 등장할 때 소급 수정을 피하려면 선행 정의가 유리한
  횡단 계약(Money·이벤트·멱등·감사)과 기계적으로 강제되는 도메인 경계를 세우고,
  모듈러 모놀리스 배선이 실제로 관통함을 검증한다. **이 Phase는 프로젝트 기반과
  아키텍처 검증에 한정하며, 어떤 실제 도메인 기능도 선점하지 않는다.**
- **학습 단위 분할 (스펙 2개)**: 결합 P0 스펙이 12 FR·8 primitive로 "1 스펙"
  상한을 넘어 학습 단위로 과다하다는 판단에 따라 P0를 2개 스펙으로 나눈다. 스펙
  번호는 프로젝트 전역 순번(NNN)이며 Phase 번호와 별개다. (표기 규칙: 마일스톤 =
  **Phase 0/1…**, 실행 단위 = **Spec 000/001…**, 스펙 내 우선순위 = **Priority
  P1/P2**. `P0.1` 같은 decimal 표기는 쓰지 않고, 하위 단계는 **Step 1/2**로만
  부른다.)
  - **`000-foundation-structure` — Step 1** — Django 모듈러 모놀리스 스캐폴딩·설정
    계층, 공개 서비스 인터페이스 패턴 + 정적 import 경계 계약·CI 강제 (VIII),
    공유 커널 패키지 위치·단방향 의존, 경계 적합성 픽스처(도메인·primitive 없음),
    문서: `docs/adr/`와 핵심 결정 ADR + state-model·glossary의 정식 경로·소유
    규칙 기록(빈 문서는 선제 생성하지 않음; 파일은 첫 상태/도메인 기능이 생성,
    XVIII·XX·거버넌스).
  - **`001-shared-kernel-primitives` — Step 2** — 공유 커널 primitive: Money(액수+통화
    구조·반올림·안분; 운영 통화 범위는 KRW 단일 — 「범위 제외」 참조, V, **test-first**), 도메인 이벤트 계약(ID·시각·집계ID + 발행/
    디스패치 분리·동기, XII·XIII, **test-first**), 멱등키 + 유니크 제약(XIII,
    **test-first**), 불변 감사 기록(필수 필드, XV-감사, **test-first**), 행위자
    역할 인가 스캐폴딩 (VI), 상관관계 식별자 기반 커널 내부 관통 추적
    (멱등→감사→이벤트 연결, XV).
  - **관통 실증의 배치(설계 원칙)**: 별도의 walking-skeleton 스펙은 두지 않는다.
    관통 검증의 커널 내부 절반(이벤트+멱등+감사가 하나의 흐름에서 상관관계
    식별자로 이어짐)은 Step 2가 직접 검증하고, 도메인 계층 절반(실제 데이터가
    서비스 인터페이스·경계 아래에서 primitive를 관통)은 **P1 첫 도메인 기능이
    겸한다** — 빈 픽스처만의 E2E 관통은 실제 도메인 관통 대비 학습·검증 가치가
    낮기 때문이다(도구 사용법 연습이 아니라 실데이터 관통이 진짜 검증이다).
- **P0에서 슬림 제거**: ACL 추상화 골격은 P0~P5에 소비자가 없어 첫 소비자
  **P6(결제)로 이관**한다(YAGNI; 헌장 IX의 "기본 원칙"은 첫 외부 호출 시 강제되며,
  P6이 ACL 공통 인터페이스+어댑터 패턴을 최초 확립).
- **선행 조건**: 없음 (헌장 1.0.0 비준 완료).
- **완료 조건** (Phase = 스펙 2개 모두 완료):
  - (Step 1) 정적 import 경계 검사가 CI에서 경계 위반 빌드를 실패시키고, (픽스처를 도메인
    계층으로 등록해) 커널 역의존이 실제로 차단되며, 결정 ADR과 state-model·
    glossary 소유 규칙이 기록된다(빈 문서 선제 생성 없음).
  - (Step 2) Money·멱등·이벤트·감사 primitive 단위 테스트가 test-first로 GREEN이고,
    커널 내부 관통(멱등→감사→이벤트가 상관관계 식별자로 연결)이 검증된다.
  - (P1 연계) 도메인 계층에서의 primitive 관통 실증은 P1 첫 도메인 기능의 완료
    조건에 포함된다(위 "관통 실증의 배치").
- **다음 Phase 연결 이유**: 이 primitive들을 나중에 도입하면 모든 기존 도메인을
  소급 수정해야 하므로 반드시 선행한다. 커널·경계·인가가 서면 그 위에 첫 번째
  **실제** 도메인(Vendor)을 격리된 앱으로 올릴 수 있다.

## Phase 1 — Vendor & Identity (Marketplace First의 구체화)

- **목적**: 원칙 III(마켓플레이스 우선)·VI(벤더 격리)를 실제 도메인으로
  구현하고, **벤더 생명주기의 상태·전이·이벤트 계약을 처음부터 정의**해 이후
  모든 도메인이 벤더 소유권·인가·상태 계약을 상속하게 한다. **P0가 검증한
  아키텍처 위에서 시작하는 첫 실제 도메인 구현.**
- **Identity 범위**: 이 Phase의 Identity는 전체 인증/멤버십/SSO 구축이 아니라,
  최소 행위자 컨텍스트(actor context)·벤더 계정 식별·서비스 경계 인가에 한정한다.
  고객 멤버십과 외부 인증 확장은 필요가 생기는 별도 스펙에서 다룬다.
- **관통 실증 겸무**: P1의 첫 도메인 기능은 **도메인 계층 primitive 관통 실증**을
  겸한다 — 실제 도메인 데이터가 서비스 인터페이스·경계 아래에서 커널 primitive
  (Money·이벤트·멱등·감사·역할·상관관계)를 관통함을 완료 조건에 포함한다
  (P0 "관통 실증의 배치" 참조).
- **구현할 핵심 도메인**:
  - 벤더 계정·소유권 엔티티 (도메인 간 참조는 ID 기반, VIII)
  - **벤더 정의·`vendor_id` 사용 범위 계약** — 벤더 = 플랫폼에 계정을 가진 입점
    파트너(테넌시·격리·생명주기의 축). `vendor_id`는 계정 식별자이며, 출고
    주체·정산 유형의 동의어(암묵 키)로 사용하지 않는다 — 판매·이행 주체의
    조합은 상품 유형에서 함수적으로 도출되지 않으므로(위탁+플랫폼 풀필먼트,
    직매입+공급사 직송 같은 조합이 실재 — 현 스코프 밖, 「범위 제외」), 거래상
    분류(위탁/직매입)는 P5가 주문 라인에 동결하는 데이터가 정본이다. ADR로
    기록 — 대안(역할별 1급 엔티티 분리, SellerOfRecord 합 타입) 비교 포함 (XX)
  - **플랫폼 1P(자사) 판매자 계정 계약** — 직매입 상품의 판매 주체는 플랫폼이
    운영하는 자사 판매자 계정이다. 카탈로그 소유·주문 분할·배송 정책은 일반
    벤더와 동일 경로를 쓰므로 "출고는 판매자 계정 기준" 규칙이 위탁/직매입
    구분 없이 균일하게 성립하며, 입점 심사 생명주기와 정산 지급(payout)은
    적용하지 않는다(가드). 이 Phase는 계정 유형 구분(1P/입점)과 가드 계약
    정의까지만 다루며, 직매입 실운영(조달)은 범위 밖이다 (「범위 제외」 참조, III)
  - 행위자 역할 모델(운영자/벤더/고객)과 **서비스 경계에서 강제되는** 벤더
    범위 인가 (VI)
  - **벤더 생명주기 상태·전이·이벤트 계약 정의** — 입점 심사 → 활성 →
    일시정지/제재 → 퇴점(wind-down)을 `docs/domain/state-model.md`에 등재
    (III). 이 Phase에서는 계약(어휘·상태·전이·이벤트)만 확정하고 ACTIVE
    경로만 실제 실행한다. **전이 오케스트레이션과 wind-down 청산 로직 구축은
    P11로 유보** (설계 먼저·구축 나중). 각 상태가 노출·주문 수용·정산 지급에
    미치는 효과는 해당 도메인 Phase(P2·P5·P10)가 이 계약을 참조해 연결한다.
  - 운영자 어드민 동작 + 행위자·사유·근거를 포함한 불변 감사 사실 (VI, XV-감사)
- **선행 조건**: P0 (인가 스캐폴딩, 감사 primitive, 서비스 인터페이스 패턴).
- **완료 조건**:
  - 벤더 B가 벤더 A의 데이터에 접근 시 서비스 경계에서 거부됨을 테스트로
    증명한다.
  - 벤더 생명주기 상태·전이·이벤트 계약이 `state-model.md`에 등재되고, 상태
    전이 이벤트 계약이 정의된다 (효과 연결은 후속 Phase, 청산 로직은 P11).
  - 운영자 개입이 감사 사실을 남긴다.
  - 벤더 소유권이 ID로 표현되어 이후 도메인이 참조할 수 있다.
  - 계정 유형(1P/입점)이 구분되고, 1P 계정에 대한 입점 심사 생명주기 전이가
    거부된다(정산 지급 비적용 가드는 계약으로 정의, 실검증은 P10). `vendor_id`
    사용 범위 계약이 ADR로 기록된다.
  - 첫 도메인 용어 도입 기능으로서 `docs/domain/glossary.md`를 최초 생성하고
    (Spec 000 소유 규칙), 벤더·1P 판매자 계정·판매 주체(seller of record)·
    위탁/직매입을 등재한다.
- **다음 Phase 연결 이유**: 소유권 주체와 상태 계약이 존재해야 Catalog·Order·
  Settlement가 각각 노출·주문 수용·지급 효과를 **처음부터** 연결한다. 벤더나
  그 상태 계약을 나중에 넣으면 카탈로그·주문·정산 전체에 소급 삽입하는
  재작업(원칙 III이 경고한 실수)이 발생한다.

## Phase 2 — Catalog

- **목적**: 벤더가 소유하는 상품·변형(variant)과 고객 노출(가시성)을
  모델링한다.
- **구현할 핵심 도메인**:
  - 상품·변형(variant), 벤더 소유 카탈로그
  - 게시/노출 상태
  - 정가(listed price) — Money 타입 (V)
  - 벤더 소유 **배송비 정책** 정의 — 기본 배송비·조건부 무료배송(주문 금액 기준)·
    **묶음배송 그룹 규칙**(같은 벤더/그룹 상품을 함께 배송하면 배송비 1회 부과).
    묶음배송 그룹은 판매자 계정 경계를 넘지 않는다 — 그룹 배송비가 복수
    판매자의 정산에 걸치지 않게 한다(교차 묶음은 「범위 제외」). 이 Phase는
    정책의 *정의*만 소관이며, 실시간 미리보기는 P4, 산정·동결은 P5,
    부분 취소 시 재평가는 P9가 담당한다 (원리 ①)
  - 상품별 **과세/면세 구분** 저장 — 가격은 부가세 포함가(내세) 원칙, 세액 계산은
    P10 정산·ERP 단계에서 역산 (원리 ①)
  - 고객용 읽기 모델 (노출 정합성 고려)
  - 노출 가격↔주문 시점 값의 우선 규칙 기초
  - **벤더 상태(SUSPENDED 등)가 카탈로그 노출에 미치는 효과 연결** — P1의
    생명주기 계약을 참조 (III)
- **선행 조건**: P1 (벤더 소유권·상태 계약), P0 (Money).
- **완료 조건**:
  - 벤더 범위로만 자기 카탈로그를 편집할 수 있다.
  - 정가가 Money로 표현된다.
  - 벤더별 배송비 정책(묶음배송 그룹 규칙 포함)과 상품 과세/면세 구분이 벤더
    범위로 관리된다.
  - 고객 읽기 모델이 노출 상태를 반영하고, 벤더 비활성 상태가 노출을 차단한다.
- **다음 Phase 연결 이유**: 팔 물건(카탈로그)이 있어야 재고·장바구니·주문이
  의미를 가진다.

## Phase 3 — Inventory & Reservations

- **목적**: 재고를 벤더 소유 자원으로 모델링하고, 원칙 XIV의 예약→확정/해제
  상태 머신을 **원자적·DB 강제** 불변식으로 구현한다 — 이 프로젝트의 동시성
  핵심 개념을 독립 학습 단위로 격리한다.
- **구현할 핵심 도메인**:
  - 재고 수량 (벤더 소유)
  - 만료 시한을 가진 되돌릴 수 있는 예약(reservation)
  - 원자적·상호 배타적 확정(confirm)/해제(release) 전이
    (`select_for_update` / 낙관적 버전 + 조건부 UPDATE / CHECK 제약)
  - 가용 수량 ≥ 0 불변식의 DB 수준 최종 보장 (XIV)
  - 만료 스위퍼의 해제와 확정 콜백의 경합 방지 (P6 결제 확정과 연결되는
    경합 케이스는 P6에서 보상 경로까지 다룸)
  - `InventoryReserved` 등 도메인 이벤트 (XII)
- **선행 조건**: P2 (상품/변형), P0 (이벤트 계약, 멱등).
- **완료 조건**:
  - 동시 예약/확정/만료 해제 경합 테스트에서 이중 판매·음수 재고가 없다.
  - 예약 만료가 관측 신호를 남긴다 (XVI 설계).
  - 예약 관련 이벤트를 발행한다.
- **다음 Phase 연결 이유**: Order가 결제 실패에 견디려면 확정 전 예약이
  필수다(XIV 근거). 예약 없이 Order를 만들면 이후 재고 정합성을 전면
  재작업해야 한다.

## Phase 4 — Cart

- **목적**: 주문 이전의 고객 장바구니를 별도 도메인으로 모델링한다 (헌장 범위
  경계: Cart를 Order에 흡수하지 않는다).
- **구현할 핵심 도메인**:
  - 고객 장바구니, 담기/빼기
  - **실시간 재계산** 가격 미리보기 (아직 동결이 아님)
  - 벤더별·묶음배송 그룹 단위 **배송비 실시간 미리보기** (P2 정책 기반, 아직
    동결이 아님)
  - 재고 가용성 best-effort 조회
  - 멀티벤더 장바구니 구성
- **선행 조건**: P2 (카탈로그·가격), P3 (가용성 조회).
- **완료 조건**:
  - 장바구니 가격이 실시간·비동결(스냅샷이 아님)임을 명시한다.
  - 멀티벤더 담기를 지원한다.
  - 주문 도메인과 분리된 앱 경계를 유지한다.
- **다음 Phase 연결 이유**: 장바구니의 "실시간·가변" 가격과 Order의 "동결·
  불변" 스냅샷이 대비되며, 원칙 IV(불변 스냅샷)의 경계선이 두 도메인 사이에서
  자연스럽게 드러난다.

## Phase 5 — Order & Checkout (가격 동결)

- **목적**: 주문 생성 시점에 재무적 사실을 불변으로 동결(IV)하고, 멀티벤더
  주문 분할과 라인별 정산/이행 분류 동결을 구현한다. **프로모션은 이 Phase에서
  decision port + no-op 정책으로 seam만 정의**하고, 역방향 흐름은 명시적으로
  범위 밖으로 선언한다 (헌장 워크플로우 게이트 요구).
- **구현할 핵심 도메인**:
  - `OrderPlaced` 상태 전이·이벤트 (XII)
  - **Pricing/Promotion decision port + no-op 정책** — 가격 계산 파이프라인이
    프로모션 결정 결과를 입력으로 받는 seam(port)을 정의하되, P5에서는 no-op
    정책(할인 0)을 주입한다. P8은 이 port에 실제 프로모션 구현체를 주입만
    하므로 총액 계산·결제 검증·스냅샷·테스트를 다시 열지 않는다 (원리 ①,
    스키마·계약 진화 expand)
  - 가격·배송비·세금(과세/면세 구분 포함) 스냅샷 동결 + 입력값 기록 (IV)
  - **배송비 산정·동결** — 벤더별 정책과 **묶음배송 그룹 단위** 산정을 벤더 단위
    주문 분할과 정합되게 동결하고, 배송비 그룹 구성(어떤 라인이 어느 그룹으로
    묶여 얼마가 부과됐는지)을 스냅샷에 기록한다. P9의 부분 취소 재평가·안분이
    이 기록을 입력으로 사용한다 (IV, 원리 ①)
  - **라인별 위탁/직매입(정산·이행) 분류를 판매 시점에 동결** — 분류 값은
    boolean이 아닌 enum으로 두어 향후 세분화(위·수탁/중개, 특정매입 등)를
    expand로 수용한다 (IV, 정산 대비, 스키마·계약 진화)
  - 멀티벤더 주문의 벤더 단위 분할 (VI)
  - **벤더 상태(SUSPENDED 등)가 신규 주문 수용에 미치는 효과 연결** — P1 계약
    참조 (III)
  - 재고 예약 소비 (P3)
  - 동결 후 미결제 유효기간·만료 시 처리(예약 해제)
  - 스냅샷 스키마 버전 표식 (스키마·계약 진화)
- **선행 조건**: P3 (예약), P2 (가격·노출), P1 (벤더 상태 계약),
  Spec 001 (Money·이벤트 계약·멱등 primitive). 스냅샷 버전 표식은 이 Phase에서
  최초 정의한다.
- **완료 조건**:
  - 주문 스냅샷이 스키마 버전 표식을 포함하고 재계산되지 않는다.
  - 가격 파이프라인이 프로모션 decision port를 경유하며, no-op 정책으로
    동작함을 테스트로 증명한다 (P8이 구현체만 주입할 수 있는 seam 확립).
  - `OrderPlaced` ≠ 결제 성공이 코드/이벤트로 분리된다.
  - 멀티벤더 분할·라인 분류가 동결되고, 정지 벤더의 신규 주문이 거절된다.
  - 묶음배송 그룹 단위 배송비 산정이 벤더 분할과 정합되게 동결되고, 그룹 구성이
    스냅샷에 기록된다.
  - 미결제 만료가 예약을 해제한다.
  - 역방향 흐름이 out-of-scope로 명시된다.
- **다음 Phase 연결 이유**: 동결된 주문 총액이 있어야 결제를 "동결값 근거로만"
  요청·검증할 수 있다(IV). 프로모션(P8)은 이미 정의된 decision port에 구현체를
  주입하므로 가격 파이프라인 재작업이 없다.

## Phase 6 — Payment (외부 ACL · 확정 · 보상 · 대사 설계)

**Phase = 마일스톤 (여러 스펙).** 예: (1) 결제 ACL·게이트웨이 원문·상관관계
식별자, (2) 결제 상태 머신·멱등·예약 확정, (3) 지연·중복 콜백·보상·UNKNOWN
대사.

- **목적**: 결제 게이트웨이를 ACL 뒤에 격리(IX)하고, 동결 총액 근거의 멱등
  결제·예약 확정·**실패/지연 시 보상**·대사 가능 설계를 구현한다. **프로젝트에서
  위험이 가장 집중된 Phase.**
- **구현할 핵심 도메인**:
  - 결제 게이트웨이 ACL — 공통 인터페이스 + 복수 PG 어댑터 (IX). **P0에서 이관:
    ACL 공통 인터페이스+어댑터 패턴을 이 Phase에서 최초 확립**
  - 동결 총액 대조 후 결제 요청 (IV)
  - 멱등키 (XIII)
  - **결제 상태 머신** — PENDING / SUCCEEDED / FAILED / **UNKNOWN** /
    **COMPENSATING** (XIV·XV)
  - `PaymentSucceeded` 시 재고 예약 원자적 확정 (XIV)
  - **결제 성공↔예약 만료 경합의 보상 경로** — 결제 콜백이 예약 해제 이후
    지연 도착하거나, 결제 성공 후 재고 확정이 실패하는 경우의 대체 정책(자동
    환불 또는 명시적 보상)을 정의 (헌장 XIV: 시한 초과 시 대체 정책 MUST)
  - **지연·중복 웹훅** 처리 (멱등·순서 무관, XIII)
  - **UNKNOWN 거래 조회·재대사** — 응답 유실 시 외부 조회로 상태 확정 (XV)
  - **트랜잭션 경계 규율** — 외부 호출은 열린 DB 트랜잭션 밖에서 실행:
    의도 기록 커밋 → 트랜잭션 밖 외부 호출 → 결과를 새 트랜잭션에 기록
  - 대사 가능 설계 — PG 거래·승인번호·금액·시각·상관관계 ID 저장, 게이트웨이
    요청·응답 원문 기록 (비밀정보·PII 마스킹·보존기간 한정) (XV-대사)
  - 자동 대사 *배치 운영*은 P12로 유보 (설계는 여기서 완료)
- **선행 조건**: P5 (동결 총액), P3 (확정할 예약), P0 (멱등·이벤트 계약; ACL
  패턴은 이 Phase에서 최초 확립).
- **완료 조건**:
  - 결제 재시도·중복 웹훅이 멱등이다.
  - 동결값과 외부 통지 금액을 대조한다.
  - 외부 호출이 열린 DB 트랜잭션 밖에서 실행된다.
  - **결제 성공 시 예약이 이미 해제된 경우, 정의된 보상 정책(자동 환불/재요청/
    거절)이 실행됨을 테스트로 증명한다.**
  - UNKNOWN 거래가 조회·재대사로 종국 상태에 도달한다.
  - 상관관계 ID·마스킹된 원문 기록이 존재한다.
- **다음 Phase 연결 이유**: 결제가 확정(또는 보상)되어야 이행(배송)이 시작될
  근거가 생기고, 대사·정산의 재무 사실(결제 원장)이 확보된다.

## Phase 7 — Fulfillment & Shipping

- **목적**: 이행·배송 상태 머신과 이벤트를 구현하고 표준 상태 모델 문서를
  확장한다.
- **구현할 핵심 도메인**:
  - 배송 상태 머신 (`ShipmentStarted` 등 이벤트, XII)
  - 벤더 범위 이행
  - 배송 추적
  - 멀티벤더 주문의 벤더별 부분 이행
  - **고객 알림 ACL(이메일 등) 최초 도입** — P6이 확립한 ACL 공통 인터페이스+
    어댑터 패턴을 재사용하는 두 번째 사례 (IX). 주문 확인·배송 시작 등 고객
    통지를 도메인 이벤트 소비로 발송한다. 알림은 대사할 재무·재고 값이 없어
    XV-대사 비대상이며(헌장 XV), best-effort 채널로 시작하되 반드시 전달되어야
    하는 통지의 신뢰 전달 전환은 P12 아웃박스 스펙에서 판단한다 (XIII)
  - `docs/domain/state-model.md`에 배송 상태·전이 등재 (거버넌스 표준 상태 모델)
- **선행 조건**: P6 (확정 결제, ACL 패턴), P5 (주문 라인).
- **완료 조건**:
  - 배송 상태 전이가 표준 상태 문서에 등재된다.
  - 상태 전이가 이벤트·관측 신호를 발행한다.
  - 벤더별 부분 이행을 지원한다.
  - 주문 확인·배송 시작 통지가 알림 ACL을 통해 발송된다(도메인 코드의 직접
    이메일/SMS 호출 0건).
- **다음 Phase 연결 이유**: 정방향 흐름(주문 → 결제 → 이행)이 완성되어야
  역방향 흐름(취소/반품/환불)이 참조할 "이미 일어난 사실"이 존재한다. 프로모션도
  완성된 파이프라인 위에 얹기 쉬워진다.

---

## ★ 중간 마일스톤 M1 — 내부 베타 (P7 완료 후)

Phase가 아닌 **검증 마일스톤**이다(새 도메인 구현 없음).

- **목적**: 정방향 척추(카탈로그 → 주문 → 결제 → 이행)를 실제 사용 흐름으로 조기
  검증한다. 특히 위험이 가장 집중된 결제(P6) 설계가 실사용 없이 P12까지
  검증되지 않은 채 방치되는 것을 막는다.
- **범위**: PG **테스트 모드(가상 결제)** 로 내부 사용자만 카탈로그 탐색 →
  장바구니 → 주문 → 가상 결제 → 배송 추적 전 과정을 수행한다. **실거래가
  아니므로** 헌장 go-live 게이트(실사용자·실거래 진입)의 대상이 아니다. 실거래는
  환불(P9) 없이는 법적으로 불가하다(전자상거래법상 청약철회 의무) — 그래서 이
  마일스톤은 가상 거래에 한정하고, 실거래 검증은 M2로 미룬다.
- **완료 조건**:
  - 내부 사용자가 전 과정을 운영자 개입 없이 완주한다.
  - 발견된 설계 결함이 해당 스펙 개정(또는 후속 Phase 스펙의 입력)으로 환류된다.

## Phase 8 — Promotion (Order 이후 · decision port에 구현체 주입)

- **목적**: 프로모션을 1급 도메인으로 구현(X)하고, **P5에서 정의한 Pricing/
  Promotion decision port에 실제 구현체를 주입**한다(no-op → 실제 정책). 포인트는
  "결정은 내부·원장·자원은 외부"(IX·XIV)라는 미묘한 경계를 학습한다.
- **구현할 핵심 도메인**:
  - 프로모션 도메인 (캠페인·쿠폰·바우처·할인 정책)
  - **P5 decision port 구현체 주입** — 가격 파이프라인은 그대로 두고 정책만
    교체하므로 총액·결제·스냅샷 계약이 불변 (원리 ①)
  - **4대 정책 축** (X):
    1. 중복 적용(stacking) 허용 조합·우선순위
    2. 계산 적용 순서 (정률/정액)
    3. 재원 부담 주체 (플랫폼 vs 벤더)와 정산 반영 방식
    4. 한도·예산 소진 시 동작
  - Order 스냅샷의 가격 동결 입력값 확장 (스키마 버전 expand, IV)
  - 포인트 적립/사용 **결정(내부)** + 포인트 관리 ACL (외부 SoR·원장, IX)
  - **포인트 사용의 예약→확정/해제 흐름** — 외부 포인트 잔액을 사용할 때 결제
    전 예약, 결제 성공 시 확정, 실패·만료 시 해제 (XIV, P3 재고 예약·P6 결제
    확정과 동일한 패턴으로 연결)
  - **포인트 대사·감사** — 포인트 거래 ID·상관관계 식별자, 멱등키, 게이트웨이
    원문 감사 기록, 느린 외부 실행의 timeout fallback 정책 (XV, XIV)
  - 예산 자원의 DB 불변식 (XIV)
- **선행 조건**: P5 (decision port·동결 파이프라인·스냅샷 버전), P6 (결제
  확정/보상 패턴과 ACL 패턴), Spec 001 (Money·이벤트·멱등·감사 primitive).
- **완료 조건**:
  - 스택킹·계산 순서·재원 부담·한도 정책이 스펙에 명시된다 (X 게이트).
  - 프로모션 결과가 주문 스냅샷에 동결되고, P5 가격 파이프라인은 수정 없이
    구현체 주입만으로 동작한다.
  - 포인트 결정(내부)과 잔액·자원(외부 원장)이 분리되고, 포인트 사용이
    예약→확정/해제로 처리되며 timeout fallback이 정의된다.
  - 포인트 이동이 포인트 거래 ID·상관관계 ID로 대사 가능하다.
  - **재원 귀속이 정산 대비로 기록된다.**
- **다음 Phase 연결 이유**: 프로모션 재원 귀속(누가 부담하는가)이 확정돼야
  역방향 흐름의 혜택 복원/회수와 정산 안분(pro-rata)이 정의 가능하다.

## Phase 9 — Reverse Flows (취소·반품·교환·환불)

**Phase = 마일스톤 (여러 스펙).** 예: (1) 취소, (2) 반품·교환, (3) 환불 실행,
(4) 혜택 복원·정산 조정. 모두 append-only·부분 범위 원칙을 공유한다.

- **목적**: 역방향 흐름을 1급 비즈니스 흐름으로 구현한다(XI) — 원본 스냅샷을
  참조하는 append-only 재무 사실로 기록한다.
- **구현할 핵심 도메인**:
  - 취소/반품/교환/환불 흐름
  - **append-only** 역방향 재무 사실 (원본 불변, IV·XI)
  - 부분 범위 (라인 아이템 단위, 멀티벤더 주문의 벤더 단위)
  - 혜택(쿠폰·포인트·프로모션) 복원/회수 정책 (만료된 혜택 포함) — 포인트는
    P8의 포인트 예약/확정/대사 흐름과 정합
  - 주문 단위 할인·배송비의 라인/벤더 안분(pro-rata)과 끝전(반올림 잔여) 귀속
  - **묶음배송 그룹의 부분 취소 재평가** — 그룹 일부 취소로 무료배송 조건이
    깨지거나 그룹이 분리될 때의 배송비 재산정과 고객 고지·부담 정책 (P5가 동결한
    배송비 그룹 기록이 입력)
  - 부분 취소로 프로모션 조건이 깨질 때의 재평가
  - 환불 수단·순서
  - **플랫폼 = seller of record 환불 의무** (벤더 지급능력·협조와 무관, XI)
  - 벤더 귀책 환불의 정산 원장 append 차감 사실 (P10 대비)
- **선행 조건**: P6 (환불할 결제·보상 경로), P7 (반품할 이행), P8 (복원할
  혜택·재원 귀속·포인트 예약 흐름).
- **완료 조건**:
  - 원본 스냅샷 미수정·새 사실 append를 검증한다.
  - 부분 라인/벤더 취소를 지원한다.
  - 혜택 복원/회수·안분·프로모션 재평가·환불 순서를 명시한다 (XI 게이트).
  - 묶음배송 그룹의 부분 취소 시 배송비 재평가 정책이 명시·검증된다.
  - 벤더 귀책분이 정산 차감 사실로 기록된다.
- **다음 Phase 연결 이유**: 정방향·역방향 재무 사실이 모두 append-only로
  쌓이면, 정산이 이를 집계·상계할 수 있는 완전한 원장 소스를 갖게 된다.

## ★ 중간 마일스톤 M2 — 제한 파일럿 (P9 완료 후)

Phase가 아닌 **검증 마일스톤**이다(새 도메인 구현 없음).

- **목적**: 소수 벤더·소액 **실거래**로 결제·역방향 흐름(취소/환불)을 실환경에서
  검증한다. M1(가상 결제)이 검증하지 못하는 실PG 웹훅·지연·실환불을 만난다.
- **범위**: 소수 협력 벤더, 소액·저위험 상품, 실PG 결제 + 취소/환불 경로 가동.
  정산 도메인(P10)이 아직 없으므로 파일럿 기간의 벤더 지급은 **파일럿 한정 수동
  정산**으로 운영한다 — 이는 헌장 XV가 금지하는 "정상 운영의 수기 *재처리* 의존"
  (놓친 건의 수기 복구)이 아니라, 아직 구축 전인 도메인의 한시적 대체 절차이며
  종료 시점(P10 완료)을 명시한다.
- **선행 게이트 (헌장 정합성 주의)**: 헌장 go-live 게이트는 규모와 무관하게
  **실거래 진입**에 적용된다. 따라서 M2 전에 파일럿이 행사하는 경계에 한정한
  운영 인프라 — 결제(P6) 경계의 자동·주기 대사 배치(XV-대사)와 능동 알림 채널·
  탐지(XVI) — 를 P12에서 **이 범위만 앞당겨** 가동하고, 사람이 명시적으로
  승인해야 한다(XIX). P12는 이를 전체 연동 경계로 확장·완성한다.
- **완료 조건**:
  - 실거래 주문·취소/환불이 무결하게 처리되고 대사 배치가 파일럿 거래를 자동
    대사한다.
  - 발견 결함이 P10~P12 스펙의 입력으로 환류된다.
  - 파일럿 수동 정산의 대상·기간·종료 조건이 기록된다.

## Phase 10 — Settlement

**Phase = 마일스톤 (여러 스펙).** 예: (1) 정산 원장·계산(위탁/직매입·재원 귀속),
(2) 3자 대사·매칭, (3) 보류·상계·지급(payout)·ERP 연동.

- **목적**: 플랫폼·PG·벤더 3자 대사 가능 정산을 구현한다(XV) — 거의 모든 선행
  도메인의 재무 사실이 수렴하는 지점.
- **구현할 핵심 도메인**:
  - 3자(플랫폼·PG·벤더) 대사 정산 (XV)
  - 위탁 vs 직매입 분류 기반 정산 (P5 동결값) — 직매입 라인은 벤더 지급
    대상에서 제외하고 플랫폼 매출로 귀속한다. 공급사 매입 대금 지급은 수수료
    정산과 성격이 다른 조달 흐름이므로 이 도메인의 범위 밖이다 (「범위 제외」
    참조)
  - 과세/면세 구분 기반 **세액 집계** — 부가세 포함가에서 세액을 역산해
    세금계산서·ERP 전표에 대비 (P2 구분·P5 동결값)
  - 프로모션 재원 귀속 반영 (P8)
  - append-only 벤더 정산 원장
  - **지급 보류(hold) 1급 개념** (XI)
  - 벤더 귀책 환불 상계·마이너스 잔액·차기 이월·미회수 채권 (XI)
  - 지급(payout)
  - **벤더 상태(WINDING_DOWN 등)가 정산 지급에 미치는 효과 연결** — P1 계약
    참조 (III)
  - ERP 연동 ACL (IX)
  - 자동 정산·대사 *배치 운영*은 P12로 유보
- **선행 조건**: P5 (라인 분류), P6 (결제 원장), P8 (재원 귀속),
  P9 (역방향 차감), P1 (벤더 상태 계약).
- **완료 조건**:
  - 3자 대사 조인 키·상관관계 ID를 확보한다.
  - 위탁/직매입·재원 귀속이 정산에 정확히 반영된다.
  - 보류·상계·마이너스 잔액·이월을 모델링한다.
  - append-only 원장을 유지한다.
- **다음 Phase 연결 이유**: 정산 지급 효과가 정의돼야 벤더 생명주기(정지·퇴점)의
  wind-down 청산을 완결할 수 있다.

## Phase 11 — Vendor Lifecycle (wind-down 구축 + 통합 검증)

- **목적**: P1에서 *계약*으로 정의하고 P2·P5·P10에서 *효과*를 연결한 벤더
  생명주기를, **전이 오케스트레이션과 wind-down 청산 로직으로 구축·통합
  검증**한다(III). (상태 계약 자체는 P1에서 이미 정의됨 — 이 Phase는 retrofit이
  아니라 완결·통합.)
- **구현할 핵심 도메인**:
  - 입점 심사 → 활성 → 일시정지/제재 → 퇴점 FSM의 전이 오케스트레이션 —
    1P 자사 계정은 비대상(P1 가드) (III)
  - 각 상태 효과의 **통합 검증** — 노출 차단(P2)·주문 거절(P5)·지급 보류(P10)가
    상태 전이와 일관되게 동작함을 end-to-end로 확인
  - **퇴점 = 즉시 삭제가 아닌 wind-down 청산** (진행 중 주문·미정산 금액·법정
    반품 기간 등 잔여 의무를 청산한 뒤에만 퇴점 완료)
  - 상태 전이 이벤트(XII)·관측 알림(XVI)
  - 운영자 개입 어드민 동작·감사 (VI)
- **선행 조건**: P1 (상태 계약), P2 (노출 효과), P5 (주문 수용 효과),
  P10 (정산 지급 효과), P9 (반품 기간 청산).
- **완료 조건**:
  - 전이 오케스트레이션이 P1 계약대로 동작하고 표준 상태 문서와 일치한다.
  - 각 상태 효과가 세 도메인(노출·주문·정산)에서 통합 검증된다.
  - wind-down이 잔여 의무 청산 후에만 퇴점을 완료한다.
  - 전이가 이벤트·알림을 발행한다.
- **다음 Phase 연결 이유**: 도메인 척추가 완성된다. 남은 것은 이 전체를
  실사용자·실거래에 견디게 하는 운영 인프라뿐이다.

## Phase 12 — Operational Hardening & Go-Live

**Phase = 마일스톤 (여러 스펙 + ops 트랙).** 예: (1) 대사 배치·알림 운영,
(2) 아웃박스·비동기 릴레이·DLQ, (3) 운영 준비(ops readiness) 트랙.

- **목적**: 헌장이 "go-live 전 MUST"로 단계화한 운영 인프라를 실제로 가동하고
  (XV·XVI 운영, XIII 신뢰 전달), 실제 운영 가능성을 위한 운영 준비를 완료한다.
- **구현할 핵심 도메인**:
  - 연동 지점별 자동·주기 **대사 배치** (놓친 건을 스스로 따라잡음, XV)
  - 능동 알림 채널·**time-to-detect** 지표 운영 (XVI)
  - **프로세스 경계를 넘거나 외부 부수효과가 있는 이벤트만** 트랜잭셔널
    아웃박스 + 비동기 릴레이로 전환 (XIII). **동일 트랜잭션 내부 동기 핸들러는
    그대로 유지**한다 (헌장 XIII: 동기 핸들러 면제 — 전면 재설계가 아니라 필요
    경계만 전환). P12는 재설계보다 **운영 활성화·재처리·DLQ 검증**에 집중.
  - poison-message 격리·알림 (XIII·XVI)
  - **Go-live 게이트 체크리스트** — 트리거된 XV-대사·XVI별 운영 인프라 가동
    확인, 사람의 명시적 승인 (XIX)
  - **운영 준비(ops readiness) 트랙**:
    - *헌장 근거 O (MUST)*: 시크릿 관리, PII 마스킹·파기 검증(IV·XV), 보안·
      인가 감사(VI)
    - *헌장 밖·일반 운영 준비 (모델링 도메인 스펙이 아닌 ops 항목)*: 백업/복구
      드릴, 마이그레이션·롤백 리허설, 부하·용량 검증, 운영 런북·장애 대응 훈련.
      이들은 도메인 스펙이 아니라 운영 준비 항목으로 별도 추적하되 go-live
      게이트에 포함한다.
- **선행 조건**: 대사·관측 신호를 첫날부터 설계해 둔 P3·P6·P8·P10·P11.
- **완료 조건**:
  - 자동 대사 배치가 수기 재처리 없이 놓친 건을 따라잡는다 (XV).
  - time-to-detect가 1급 품질 지표로 관리된다 (XVI).
  - 프로세스 경계/외부 부수효과 이벤트가 아웃박스로 최소 1회 전달되고, 내부
    동기 핸들러는 기존 동작을 유지한다 (XIII).
  - 시크릿·PII·인가 검증이 통과한다 (IV·VI·XV).
  - ops readiness 항목(백업/복구·롤백·부하·런북)이 go-live 게이트에서 확인된다.
  - go-live 게이트를 사람이 명시적으로 승인한다.
- **다음 Phase 연결 이유**: 실사용자·실거래 진입(go-live). 이후 확장(신규 외부
  연동, 추가 도메인 등)은 동일한 spec → plan → tasks → impl 사이클로 이 척추
  위에 얹는다.

## 로드맵 운영 노트

### 공통 Definition of Done

모든 Phase·스펙은 다음을 충족해야 완료로 본다 (헌장 I·XVII·XIX·거버넌스):

- **스펙·계획의 사람 승인**: 작성 세션과 분리된 별도 세션(또는 다른 모델)의
  적대적 리뷰 결과를 스펙/plan에 기록한 뒤 사람이 명시적으로 승인 (I·XIX,
  거버넌스 승인 주체).
- **2계층 Constitution Check**: Phase 0 리서치 전 + Phase 1 설계 후 게이트를
  통과 (코어 게이트 항상 점검 + 트리거된 원칙 조건부 점검). 일탈은 Complexity
  Tracking 표에 근거와 함께 기록.
- **Tasks traceability**: 트리거된 원칙 → Task ID 추적이 tasks.md에 존재.
- **RED/GREEN 테스트 증거**: 관찰 가능한 동작은 test-first(RED → GREEN)로
  증명 (foundational 작업 포함, 순수 스캐폴딩만 예외). 레거시 특성화 테스트는
  통과 상태로 시작 가능 (XVII).
- **경계 강제 통과**: 정적 import 경계 계약(도구는 plan에서 선택)이 CI에서 통과 (VIII).
- **ADR 기록**: 새 도메인·새 외부 연동·새 도메인 간 인터페이스는 `docs/adr/`에
  ADR로 남긴다 (XX, SHOULD).

### 기타

- **경험적 가지치기 (거버넌스)**: 기능 3개 완료마다 Constitution Check 결과를
  집계한다. P0~P2 완료 후 1차 점검이 원칙 20 → 13~14개 병합 근거 데이터의 첫
  수확 지점이다.
- **Cart(P4)의 위치**: 카탈로그 직후·주문 직전에 두어 "실시간 가격 vs 동결
  스냅샷" 대비를 학습 장치로 활용한다. 매우 얇게 유지할 수 있으나 Order에
  흡수하지 않는다 (헌장 범위 경계).
- **미목록 도메인**: 고객/멤버십(Customer/Membership)은 P1의 고객 행위자에서
  최소로 출발해, 필요가 생기는 Phase에서 별도 스펙으로 확장한다 (Order에 암묵
  흡수 금지, 헌장 범위 경계).
- **범위 제외**: 광고·수익화(Ads)는 헌장 1.0.0에 도메인으로 존재하지 않으므로
  이 로드맵의 범위 밖이다. 필요가 생기면 별도 헌장 개정과 로드맵으로 다룬다.
- **범위 제외 (조달·직매입 실운영)**: 공급사(supplier) 관리·발주(PO)·입고·매입
  원가·매입 대금 지급 등 조달(procurement) 도메인은 범위 밖이다. 이 로드맵의
  직매입 지원은 (a) 플랫폼 1P 판매자 계정 계약(P1), (b) 라인별 위탁/직매입
  분류 동결(P5), (c) 직매입 라인의 지급 제외·매출 귀속(P10)까지로 한정한다.
  매입 대금 지급은 수수료 정산과 성격이 다르므로 정산(P10)에 편입하지 않으며,
  직매입을 실제 사업으로 운영하는 시점에 별도 Phase(로드맵 개정)로 덧입힌다.
- **범위 제외 (도메인 외 항목)** — 암묵적 공백을 남기지 않기 위해 명시한다(헌장
  범위 경계: 의도적 범위 밖은 명시적으로 선언):
  - **멀티통화**: Money 타입은 액수+통화 구조를 유지하지만(V), 이 로드맵의 운영
    통화 범위는 **KRW 단일**이다. 통화 추가는 통화별 정밀도·반올림 규칙을 다루는
    별도 스펙으로 한다.
  - **세율 엔진**: 세금은 부가세 포함가(내세) + 상품별 과세/면세 구분(P2 저장·
    P5 동결·P10 세액 역산)으로 한정한다. 복합 세율·국가별 세제·별도 세액 계산
    엔진은 범위 밖이다.
  - **검색·탐색(Discovery)**: P2의 고객용 읽기 모델은 목록·노출 정합성까지만
    다룬다. 전문 검색(full-text)·추천·카테고리 고도화는 범위 밖이며, 필요 시
    별도 스펙으로 다룬다.
  - **풀필먼트 서비스·다중 출고지**: 이행 주체는 판매자 계정 범위로 한정한다
    (P7). 판매자와 분리된 이행 주체(플랫폼 풀필먼트·3PL)와 출고지(창고)
    모델링은 범위 밖이며, 필요 시 별도 스펙으로 다룬다.
  - **판매자 교차 묶음배송**: 묶음배송 그룹은 판매자 계정(정산 경계)을 넘지
    않는다(P2 정책·P5 동결). 교차 묶음은 그룹 배송비의 복수 판매자 안분·부담
    주체 모델링이 선행되어야 하므로 범위 밖이다.

### 학습 가이드 (난이도·선수 지식·학습 성과)

이 로드맵을 강의·스터디 자료로 쓸 때의 보조 표다. 난이도는 절대 시간이 아닌
**상대 표기**(●1~5)다 — 학습자의 배경에 따라 소요는 크게 달라지므로 시간
추정 대신 "어디서 벽을 만나는지"를 알린다. 선수 지식은 해당 Phase **시작 전에**
따로 학습해 두면 좋은 주제다.

| Phase | 난이도 | 미리 갖추면 좋은 지식 | 이 Phase를 마치면 할 수 있는 것 (학습 성과) |
|---|---|---|---|
| P0 | ●●○○○ | Django 앱 구조, Python import 체계 | 도메인 경계를 도구로 강제하는 모듈러 모놀리스 골격을 세우고 "왜 경계가 먼저인가"를 설명한다 |
| P1 | ●●○○○ | 인증(authentication) vs 인가(authorization) 구분 | 멀티테넌트 소유권과 역할 인가를 서비스 경계에서 강제하고, 상태 계약을 "설계 먼저"로 정의한다 |
| P2 | ●●○○○ | Django 모델링 기초 | 벤더 소유 카탈로그·배송비 정책·노출 규칙을 벤더 범위로 모델링한다 |
| P3 | ●●●●○ | **DB 트랜잭션·격리 수준·행 잠금(`select_for_update`)** | 경합 조건(이중 판매)을 재현하는 테스트를 작성하고 DB 수준 불변식으로 차단한다 |
| P4 | ●○○○○ | — | "실시간·가변" 도메인과 "동결·불변" 도메인의 경계를 설명한다 |
| P5 | ●●●●○ | 스냅샷·이벤트 개념(P0), port/seam 개념 | 재무 사실 동결, 멀티벤더 분할, 묶음배송 그룹 산정, no-op port 설계를 구현한다 |
| P6 | ●●●●● | **웹훅·재시도·at-least-once 전달, 멱등성** | 외부 결제의 실패·지연·중복·응답 유실을 멱등·보상·대사 설계로 다룬다 (최고 난이도 구간) |
| P7 | ●●○○○ | 상태 머신 기초 | 벤더별 부분 이행 상태 머신과 알림 ACL 재사용을 구현한다 |
| P8 | ●●●●○ | P5의 decision port 복습 | 스택킹·계산 순서·재원 부담·한도라는 프로모션 4대 정책 축을 스펙으로 결정하고 port에 주입한다 |
| P9 | ●●●●○ | 안분·반올림(Money, Spec 001) | append-only 역방향 재무 사실, 안분·끝전 귀속, 묶음배송·혜택 재평가를 설계한다 |
| P10 | ●●●●● | 원장(ledger)·상계 개념, (선택) 복식부기 감각 | 3자 대사 가능한 정산 원장과 보류·상계·이월을 모델링한다 |
| P11 | ●●●○○ | FSM 복습(P1 계약) | 생명주기 전이 오케스트레이션과 wind-down 청산을 세 도메인에 걸쳐 통합 검증한다 |
| P12 | ●●●●○ | 모니터링·알림 운영 기초 | 아웃박스·대사 배치·time-to-detect를 운영하고 go-live 게이트를 통과시킨다 |

- **동기 유지 팁**: P0는 도메인 없는 스펙 2개로 시작하는 냉각 구간이다. 각
  스펙을 시작할 때 헌장 도출 근거가 된 실제 인시던트 유형("이걸 안 하면
  P6/P10에서 무엇이 터지는가")을 티저로 함께 읽으면 동기가 유지된다. P1의 첫
  도메인 기능이 primitive가 실제 데이터로 관통하는 첫 "돌아가는 것"이다.
- **난이도 급상승 구간**: P2→P3(동시성), P5→P6(외부 실패 모드)이 두 벽이다.
  위 선수 지식(굵은 글씨)을 Phase 시작 전에 반드시 선행 학습한다.

## 부록 A — 헌장 원칙 요약표 (참조 보조)

로드맵 본문의 로마 숫자 참조를 한 곳에서 확인하기 위한 **한 줄 요약**이다.
정본은 [헌장](../.specify/memory/constitution.md)이며, 요약과 본문이 다르면
헌장 본문이 우선한다.

| 원칙 | 한 줄 요약 |
|---|---|
| I. 스펙 우선 | 승인된 스펙 없이 코드를 작성하지 않는다 (비협상) |
| II. 도메인 우선 | 프레임워크가 아니라 비즈니스 도메인이 설계를 주도한다 |
| III. 마켓플레이스 우선 | 멀티벤더가 기본형; 벤더는 생명주기 상태 머신을 가진 도메인이다 |
| IV. 불변 스냅샷 | 재무 값은 확정 시점에 동결; 정정은 수정이 아니라 새 사실 추가(append) |
| V. Money | 금액은 액수+통화 타입, 부동소수점 금지, 반올림 규칙 명시 |
| VI. 벤더 격리·인가 | 벤더 간 데이터 접근 금지를 서비스 경계에서 강제; 운영자 개입은 감사되는 어드민 동작 |
| VII. 모듈러 모놀리스 | Django 모놀리스로 시작; 마이크로서비스는 필요 입증 전 금지 |
| VIII. 도메인 경계 | 공개 인터페이스·ID 참조만 허용, 경계는 도구로 기계적 강제 |
| IX. 외부 시스템 ACL | 외부 연동은 안티코럽션 레이어 뒤에; 도메인은 공통 인터페이스에만 의존 |
| X. 프로모션 = 도메인 | 프로모션은 1급 도메인; 스택킹·순서·재원·한도 4대 정책 축 명시 |
| XI. 역방향 1급 | 취소·반품·환불은 append-only 재무 사실; 부분 범위 지원; 플랫폼이 환불 의무 주체 |
| XII. 이벤트 지향 | 중요한 상태 변화는 도메인 이벤트로 발행; 주문 생성 ≠ 결제 성공 |
| XIII. 멱등·신뢰 전달 | 부수효과 연산은 멱등키; 경계 넘는 이벤트는 아웃박스로 최소 1회 전달 |
| XIV. 예약·보상 | 미확정 자원은 만료 있는 예약으로; 불변식은 DB 수준에서 최종 보장 |
| XV. 대사·감사 | 상관관계 ID는 첫날부터; 수기 재처리로 정상 운영 금지; 불변 감사 기록 |
| XVI. 관측성 | 자동 상태 변경은 신호를 남기고 능동 알림; time-to-detect는 품질 지표 |
| XVII. 근거 기반 | "현재 운영 중"은 정확성의 근거가 아님; 주장에는 실행 관측 근거 |
| XVIII. 학습 지향 문서화 | 스펙에 비즈니스 맥락·도메인 개념·규칙·엣지 케이스 필수 |

## 부록 B — 학습 보조 용어표 (비공식)

> **주의**: 이 표는 로드맵 독해를 돕는 **비공식 학습 보조 자료**다. 정식
> 용어집은 `docs/domain/glossary.md`이며, 소유 규칙(Spec 000 BR-5)에 따라 첫
> 도메인 용어를 도입하는 기능이 최초 생성한다. **수명 규칙**: 정식 용어집이
> 어떤 용어를 다루기 시작하면 이 표에서 그 행의 정의를 삭제하고 정식 용어집
> 링크로 대체한다 — 정의의 출처는 항상 한 곳이다. 이 표는 정식 용어집이 아직
> 다루지 않는 용어의 임시 독해 보조로만 남으며, 이관 전에 두 문서가 다르면
> 정식 용어집이 우선한다.

| 용어 | 뜻 |
|---|---|
| ACL (안티코럽션 레이어) | 외부 시스템의 개념·API가 내부 도메인을 오염시키지 않도록 사이에 두는 번역 계층 |
| port / seam | 나중에 구현을 갈아끼울 수 있도록 미리 정의해 둔 접합부(인터페이스). P5의 프로모션 decision port가 예 |
| FSM (유한 상태 머신) | 허용된 상태와 전이만 정의한 모델. 벤더 생명주기·결제·배송 상태가 예 |
| 멱등성 (idempotency) | 같은 요청이 두 번 와도 결과가 한 번 처리한 것과 같은 성질. 결제 재시도·웹훅 중복의 방어책 |
| 예약 (reservation) | 확정 전 자원을 만료 시한과 함께 잡아두는 것. 확정(confirm) 또는 해제(release)로만 끝난다 |
| 스냅샷 동결 | 주문 시점의 가격·할인·배송비 등을 그 시점 값 그대로 굳혀 다시 계산하지 않는 것 |
| append-only | 기존 기록을 고치지 않고 새 기록을 추가만 하는 방식. 환불은 원본 수정이 아니라 새 사실 추가 |
| 원장 (ledger) | 돈의 증감을 빠짐없이 순서대로 적는 장부. 정산의 근거 |
| 대사 (reconciliation) | 우리 기록과 외부(PG·포인트·ERP) 기록을 대조해 불일치를 찾는 것 |
| 안분 (pro-rata) | 주문 전체에 걸린 할인·배송비를 라인/벤더별로 비례 배분하는 것 |
| 끝전 | 안분에서 나누어떨어지지 않는 반올림 잔여 금액. 누가 갖는지 정해야 한다 |
| 위탁 / 직매입 | 위탁 = 벤더 상품을 대신 팔고 수수료를 받음 — 이 로드맵에서는 위·수탁과 중개(입점)를 포괄하는 업계 관용 용법; 직매입 = 플랫폼이 사서 되팖. 정산 방식이 다르다. 법적 구분(대규모유통업법: 직매입/특정매입/위·수탁/임대차)은 더 촘촘하나 현 스코프에서는 세분화하지 않는다 |
| 재원 부담 (귀속) | 할인 비용을 플랫폼과 벤더 중 누가 부담하는가. 정산액을 좌우한다 |
| 아웃박스 (outbox) | 이벤트를 DB 같은 트랜잭션에 먼저 저장하고 별도 프로세스가 전달해 유실을 막는 패턴 |
| walking skeleton | 도메인 로직 없이 모든 배선(경계·이벤트·인가 등)이 관통함을 보이는 최소 E2E 골격 |
| wind-down | 벤더 퇴점 시 즉시 삭제가 아니라 진행 중 주문·미정산·반품 기간을 청산한 뒤 종료하는 절차 |
| seller of record | 결제상 판매자로 표시되는 주체. 플랫폼이면 벤더와 무관하게 환불 의무를 진다 |
| UNKNOWN (결제) | 성공도 실패도 확인 못 한 결제 상태. 외부 조회·재대사로 종국 상태에 보내야 한다 |
| time-to-detect | 이상 발생부터 담당자 인지까지 걸린 시간. 관측성의 1급 품질 지표 |
